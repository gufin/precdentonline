# precedent-online-site

Веб-приложение для поиска юридических прецедентов с поддержкой семантического и классического поиска.

## Требования

- Node.js (версия 18 или выше)
- npm или yarn

## Установка

1. Клонируйте репозиторий или перейдите в директорию проекта:
```bash
cd precedent-online-site
```

2. Установите зависимости:
```bash
npm install
```

## Настройка окружения

Создайте файл `.env` в корне проекта со следующими переменными:

```env
# API ключ для классического поиска (обязательно)
API_KEY=your_api_key_here

# Базовый URL API (опционально, по умолчанию https://api.you-right.ru/gas)
API_BASE=https://api.you-right.ru/gas

# Настройки семантического поиска
SEMANTIC_API_BASE=https://semsearch.ru
SEMANTIC_API_KEY=3a6fe05a871834862d13c3497a5df8c273e50e5d0aa67d8f0a7ef05a013ce93b

# Настройки AI-анализа судебных решений
RECOGNITION_API_BASE=https://dev.you-right.ru/ur_ai_api/hs/RecognitionService
RECOGNITION_API_TOKEN=Bearer your_token_here

# Gemini API ключ (если используется)
GEMINI_API_KEY=your_gemini_key_here

# Порт для Express сервера (опционально, по умолчанию 3001)
PORT=3001
```

**Примечание:** Если файл `.env` отсутствует, приложение будет использовать значения по умолчанию из `server.js`.

## Запуск проекта

### 🐳 Вариант 1: Разработка с Docker (рекомендуется)

Docker обеспечивает консистентное окружение разработки для всей команды и упрощает онбординг новых разработчиков.

#### Требования
- [Docker](https://www.docker.com/get-started) (версия 20.10 или выше)
- [Docker Compose](https://docs.docker.com/compose/install/) (обычно включен в Docker Desktop)

#### Быстрый старт

1. Создайте файл `.env` в корне проекта (см. раздел "Настройка окружения" выше)

2. Запустите приложение одной командой:
```bash
docker-compose up
```

Приложение автоматически:
- Соберёт Docker образ с Node.js 18
- Установит все зависимости
- Запустит frontend и backend серверы
- Настроит hot-reload для автоматической перезагрузки при изменениях

3. Откройте браузер:
   - **Frontend:** http://localhost:5173
   - **Backend API:** http://localhost:3001

#### Управление контейнером

Запуск в фоновом режиме:
```bash
docker-compose up -d
```

Остановка контейнера:
```bash
docker-compose down
```

Просмотр логов:
```bash
docker-compose logs -f
```

Проверка статуса:
```bash
docker-compose ps
```

Пересборка образа после изменения зависимостей:
```bash
docker-compose up --build
```

Очистка и перезапуск:
```bash
docker-compose down -v  # Удаляет volumes
docker-compose up --build
```

#### Hot Reload

Все изменения в исходном коде автоматически отслеживаются и применяются:
- **Frontend (React/TypeScript):** Vite HMR автоматически перезагружает страницу
- **Backend (Express):** Сервер автоматически перезапускается при изменениях

#### Troubleshooting

**Проблема: Порты уже заняты**
```
Error: bind: address already in use
```
Решение: Освободите порты 3001 и 5173, или измените их в `docker-compose.yml`:
```yaml
ports:
  - "3002:3001"  # Изменить первое число (порт хоста)
  - "5174:5173"
```

**Проблема: Медленная работа на macOS/Windows**

Docker volumes могут быть медленными на macOS/Windows. Используется named volume для `node_modules`, что минимизирует проблему.

**Проблема: Контейнер не стартует**
```bash
# Проверьте логи
docker-compose logs

# Пересоберите образ
docker-compose down
docker-compose up --build
```

**Проблема: Изменения не применяются**

Убедитесь, что:
1. Контейнер запущен (`docker-compose ps`)
2. Файлы не находятся в `.dockerignore`
3. Для изменений в `package.json` нужна пересборка: `docker-compose up --build`

**Fallback: использование npm напрямую**

Если Docker вызывает проблемы, всегда можно использовать традиционный способ (см. Вариант 2 ниже).

## Деплой на сервер

Проект поддерживает два способа деплоя: автоматический через GitHub Actions (рекомендуется) и ручной через локальный скрипт.

### 🚀 Автоматический деплой через GitHub Actions (рекомендуется)

При каждом push в ветку `main` проект автоматически собирается и деплоится на FTP сервер через GitHub Actions.

#### Первоначальная настройка GitHub Actions

**Шаг 1: Настройка GitHub Secrets**

Необходимо один раз настроить секреты в GitHub репозитории:

1. Перейдите в **Settings** → **Secrets and variables** → **Actions**
2. Добавьте следующие secrets (нажмите **New repository secret** для каждого):
   - `FTP_HOST`: `server295.hosting.reg.ru`
   - `FTP_USERNAME`: ваш FTP логин
   - `FTP_PASSWORD`: ваш FTP пароль
   - `FTP_SERVER_DIR`: `/` (или другая директория)

📖 **Детальная инструкция:** См. [`.github/SECRETS.md`](.github/SECRETS.md)

**Шаг 2: Готово!**

После настройки secrets деплой будет происходить автоматически при каждом push в `main`.

#### Как работает автодеплой

```
git push origin main
    ↓
GitHub Actions запускается автоматически
    ↓
1. Установка Node.js 18
2. Установка зависимостей (npm ci)
3. Сборка проекта (npm run build)
4. Загрузка файлов на FTP
    ↓
✅ Сайт обновлен на https://precedent.online
```

#### Мониторинг деплоя

1. Перейдите на вкладку **Actions** в GitHub
2. Вы увидите workflow "Deploy to FTP"
3. Кликните на него чтобы посмотреть детальные логи
4. GitHub отправит email уведомление о результате деплоя

#### Что деплоится

- ✅ Только собранные файлы из `dist/` директории
- ✅ Инкрементальная загрузка (только измененные файлы)
- ❌ Исходный код, node_modules, .git не загружаются

### 💻 Локальный ручной деплой (fallback)

Если нужен экстренный hotfix или GitHub Actions недоступен, можно задеплоить вручную.

#### Первоначальная настройка

1. Убедитесь, что установлен `lftp`:
```bash
brew install lftp
```

2. Создайте файл `.env.deploy` на основе `.env.deploy.example`:
```bash
cp .env.deploy.example .env.deploy
```

3. Отредактируйте `.env.deploy` и укажите свои данные FTP:
```env
FTP_HOST=server295.hosting.reg.ru
FTP_USER=ваш_пользователь
FTP_PASSWORD=ваш_пароль
FTP_DIR=/
```

**⚠️ Важно:** Файл `.env.deploy` добавлен в `.gitignore` и не будет закоммичен в репозиторий.

#### Ручной деплой

Для деплоя проекта выполните одну из команд:

**Вариант 1: Собрать и задеплоить:**
```bash
npm run deploy
```

**Вариант 2: Только деплой (если уже собрали):**
```bash
./deploy.sh
```

#### Что происходит при локальном деплое

1. ✅ Проверка наличия `lftp`
2. ✅ Загрузка настроек из `.env.deploy`
3. ✅ Сборка проекта (`npm run build`)
4. ✅ Создание бэкапа текущей версии на сервере
5. ✅ Загрузка файлов на FTP
6. ✅ Проверка успешности деплоя

#### Бэкапы

При локальном деплое на сервере создается бэкап предыдущей версии с именем `backup-YYYYMMDD-HHMMSS`. Это позволяет откатиться к предыдущей версии в случае проблем.

**Примечание:** GitHub Actions деплой не создает бэкапы автоматически, файлы перезаписываются инкрементально.

#### Откат к предыдущей версии

Если нужно откатить изменения, подключитесь к FTP и восстановите файлы из бэкапа:

```bash
lftp -u user,password ftp://server295.hosting.reg.ru
cd /
mirror backup-20260116-120000 /  # замените на имя нужного бэкапа
bye
```

### Troubleshooting

#### GitHub Actions деплой не работает

**Проблема:** Workflow падает с ошибкой "Secrets not found"

**Решение:**
1. Проверьте что все 4 секрета добавлены в Settings → Secrets and variables → Actions
2. Имена секретов должны точно совпадать: `FTP_HOST`, `FTP_USERNAME`, `FTP_PASSWORD`, `FTP_SERVER_DIR`
3. Попробуйте перезапустить workflow вручную

**Проблема:** Деплой успешен, но сайт не обновился

**Решение:**
1. Проверьте логи в GitHub Actions - убедитесь что файлы загружены
2. Очистите кэш браузера (Ctrl+Shift+R или Cmd+Shift+R)
3. Проверьте что `FTP_SERVER_DIR` указывает на правильную директорию

**Проблема:** Нужен экстренный деплой, но GitHub Actions недоступен

**Решение:**
Используйте локальный ручной деплой через `./deploy.sh` (см. выше)

#### Локальный деплой не работает

**Проблема:** `lftp: command not found`

**Решение:**
```bash
# macOS
brew install lftp

# Ubuntu/Debian
sudo apt-get install lftp
```

**Проблема:** `.env.deploy` not found

**Решение:**
Создайте файл `.env.deploy` и заполните FTP credentials (см. раздел "Локальный ручной деплой" выше)

## API семантического поиска

Приложение использует внешний сервис семантического поиска для интеллектуального поиска по смыслу запроса.

### Базовая информация

- **URL:** https://semsearch.ru
- **Документация:** https://semsearch.ru/docs
- **Аутентификация:** API ключ в заголовке `X-API-Key`
- **Версия:** 0.1.0

### Доступные эндпоинты

#### 1. Семантический поиск
```
POST /api/v1/search
```

**Запрос:**
```json
{
  "query": "контрагент не оплатил поставку",
  "limit": 20,
  "min_score": 0.5,
  "filters": {
    "date_from": "2020-01-01",
    "date_to": "2025-12-31",
    "court": "Арбитражный суд города Москвы",
    "doctype": "Решение",
    "categories": ["Экономические споры"]
  }
}
```

**Ответ:**
```json
{
  "query": "контрагент не оплатил поставку",
  "took_ms": 234,
  "items": [
    {
      "document_id": "uuid",
      "case_number": "A40-12345/2023",
      "court_name": "Арбитражный суд города Москвы",
      "doc_date": "2023-05-15T00:00:00Z",
      "score": 0.87,
      "snippet": "...релевантный фрагмент...",
      "highlights": ["фрагмент 1", "фрагмент 2"],
      "url": "/cases/A40-12345-2023",
      "doctype": "Решение",
      "categories": ["Экономические споры"]
    }
  ]
}
```

#### 2. Получение полного текста дела
```
GET /api/v1/cases/{case_number}
```

#### 3. Проверка здоровья сервиса
```
GET /health
```

### Поддерживаемые фильтры

Семантический поиск поддерживает следующие фильтры:

| Фильтр | Тип | Описание | Пример |
|--------|-----|----------|--------|
| `date_from` | string | Дата от (ISO 8601: YYYY-MM-DD) | "2020-01-01" |
| `date_to` | string | Дата до (ISO 8601: YYYY-MM-DD) | "2025-12-31" |
| `court` | string | Точное совпадение названия суда | "Арбитражный суд города Москвы" |
| `doctype` | string | Тип документа | "Решение", "Постановление" |
| `categories` | string[] | Категории дел (OR условие) | ["Экономические споры", "Налоговые споры"] |

**Важно:** Формат дат для семантического API - ISO 8601 (YYYY-MM-DD), в отличие от классического API, где используется dd.mm.yyyy.

### Различия между классическим и семантическим поиском

| Параметр | Классический API | Семантический API |
|----------|------------------|-------------------|
| **URL** | https://api.you-right.ru/gas | https://semsearch.ru |
| **Метод поиска** | По ключевым словам | По смыслу запроса |
| **Фильтры в API** | Нет (только на клиенте) | date_from, date_to, court, doctype, categories |
| **Фильтры только UI** | judge, side, level, inForceOnly | Нет |
| **Score (релевантность)** | Нет | Да (0.0-1.0) |
| **Highlights** | Нет | Да |
| **Max results** | ~30 | 1-100 (настраивается) |
| **Формат дат** | dd.mm.yyyy | ISO 8601 (YYYY-MM-DD) |

## AI-анализ судебных решений

Приложение поддерживает автоматический анализ судебных решений с помощью искусственного интеллекта через Recognition API.

### Возможности AI-анализа

- **Супер-краткая фабула дела** - лаконичное описание сути дела
- **Позиции и доводы сторон** - аргументы истца и ответчика
- **Мотивировка суда** - обоснование решения суда
- **Резолютивная часть** - итог дела с суммами и действиями

### Как использовать

1. Откройте модальное окно с судебным решением (кликните на карточку результата)
2. Нажмите кнопку **"AI Анализ"** в нижней части окна
3. Дождитесь завершения анализа (обычно 10-30 секунд)
4. Изучите структурированный результат

### API Recognition Service

#### Запуск анализа

```bash
POST /api/ai-analysis/recognize
Content-Type: application/json

{
  "recognitionFunctionName": "Резюмировать решение",
  "fileData": "Полный текст судебного решения"
}

# Ответ
{
  "requestID": "e218461b-4908-4b81-816c-a12a510ec230"
}
```

#### Проверка статуса

```bash
GET /api/ai-analysis/status?requestID=e218461b-4908-4b81-816c-a12a510ec230

# Ответ (в процессе)
{
  "status": "В обработке"
}

# Ответ (завершен)
{
  "status": "Завершен",
  "result": {
    "СтрокаJson": "{...структурированный результат...}",
    "РаспознанныйТекст": "...",
    "ОписаниеОшибки": ""
  }
}
```

### Особенности работы

- **Асинхронная обработка**: Анализ происходит в фоновом режиме, система автоматически проверяет статус каждые 2-5 секунд
- **Exponential backoff**: Интервал polling увеличивается со временем для оптимизации нагрузки
- **Таймаут**: Максимальное время ожидания - 60 секунд
- **Обработка ошибок**: При ошибке доступна кнопка "Повторить попытку"
- **Безопасность**: Authorization токен хранится на backend и не раскрывается в frontend

## Устранение неполадок

### Семантический поиск недоступен

Если вы видите ошибку "Семантический поиск недоступен":
1. Убедитесь, что сервис семантического поиска запущен и доступен по адресу `https://semsearch.ru`
2. Проверьте правильность `SEMANTIC_API_KEY` в файле `.env`
3. Проверьте статус сервиса: `curl https://semsearch.ru/health`
4. Или переключитесь на классический поиск

**Проверка подключения:**
```bash
# Проверка здоровья API
curl -k https://semsearch.ru/health

# Тестовый поиск
curl -k -X POST https://semsearch.ru/api/v1/search \
  -H "Content-Type: application/json" \
  -H "X-API-Key: YOUR_KEY" \
  -d '{"query": "договор", "limit": 2}'
```

### Ошибки API

Если возникают ошибки при запросах к API:
1. Проверьте наличие и правильность `API_KEY` в файле `.env`
2. Убедитесь, что API сервер доступен по адресу, указанному в `API_BASE`
3. Проверьте логи в консоли браузера и терминале сервера
