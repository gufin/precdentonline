# Design: AI-анализ судебных решений

## Context
Приложение "Прецедент Онлайн" предоставляет доступ к базе судебных решений. Пользователи могут искать решения, но чтение полного текста занимает много времени. AI-анализ решает эту проблему, предоставляя структурированное резюме.

**Constraints:**
- Recognition API работает асинхронно (POST → polling GET)
- API требует Authorization header с Bearer token
- Время обработки может составлять от нескольких секунд до минуты
- API может вернуть ошибку или таймаут
- Необходимо скрыть API ключи от клиента

**Stakeholders:**
- Юристы - основные пользователи, нуждаются в быстром анализе
- Backend - необходим proxy для безопасности
- Frontend - должен обеспечить хороший UX во время ожидания

## Goals / Non-Goals

**Goals:**
- Предоставить структурированный анализ судебного решения за 10-30 секунд
- Обеспечить безопасное хранение API ключей (через backend proxy)
- Создать интуитивный UI с индикацией прогресса
- Обработать все возможные ошибки с понятными сообщениями
- Использовать HTTPS для всех запросов

**Non-Goals:**
- Кэширование результатов анализа (будет добавлено позже)
- Анализ нескольких решений одновременно
- Экспорт результатов анализа (будет добавлено позже)
- Настройка параметров анализа пользователем

## Decisions

### 1. Архитектура API Integration

**Decision:** Использовать backend proxy вместо прямых запросов с frontend

**Rationale:**
- API требует Authorization header с секретным токеном
- Прямые запросы с frontend раскроют токен в DevTools
- Backend proxy обеспечивает безопасность и контроль

**Implementation:**
```
Frontend → Express Proxy → Recognition API
```

### 2. Polling Strategy

**Decision:** Exponential backoff с максимальным таймаутом 60 секунд

**Rationale:**
- API возвращает статус "Завершен" когда готов результат
- Polling каждые 2 секунды вначале, затем увеличение до 5 секунд
- Максимум 20 попыток (60 секунд), затем таймаут

**Implementation:**
```typescript
// Первые 5 попыток: 2 секунды
// Следующие 10 попыток: 3 секунды
// Последние 5 попыток: 5 секунд
```

### 3. UI State Management

**Decision:** Локальное состояние в Modal.tsx с useState

**Rationale:**
- Анализ привязан к конкретному модальному окну
- Нет необходимости в глобальном состоянии
- Простота реализации и поддержки

**States:**
```typescript
type AIAnalysisState = 
  | { status: 'idle' }
  | { status: 'loading'; message: string }
  | { status: 'success'; data: AIAnalysisResult }
  | { status: 'error'; error: string };
```

### 4. Result Formatting

**Decision:** Парсить JSON из `result.СтрокаJson` и отображать в структурированном виде

**Rationale:**
- API возвращает JSON в строковом поле
- Структура содержит все необходимые разделы
- Можно красиво отформатировать с использованием Tailwind CSS

**Structure:**
```json
{
  "Супер-краткая фабула дела": "...",
  "Позиции и доводы сторон": {
    "Истец": "...",
    "Ответчик": "..."
  },
  "Мотивировка суда": "...",
  "Резолютивная часть": {
    "Победитель": "...",
    "Суммы": { ... }
  }
}
```

### 5. Error Handling

**Decision:** Graceful degradation с возможностью повторной попытки

**Rationale:**
- API может быть временно недоступен
- Network errors должны быть обработаны
- Пользователь должен иметь возможность повторить

**Error Types:**
- Network Error: "Не удалось подключиться к сервису анализа"
- Timeout: "Анализ занял слишком много времени. Попробуйте позже"
- API Error: Показать сообщение от API
- Parse Error: "Не удалось обработать результат анализа"

## API Specification

### POST /api/ai-analysis/recognize

**Request:**
```json
{
  "recognitionFunctionName": "Резюмировать решение",
  "fileData": "<полный текст решения>"
}
```

**Response:**
```json
{
  "requestID": "e218461b-4908-4b81-816c-a12a510ec230"
}
```

### GET /api/ai-analysis/status?requestID={id}

**Response (в процессе):**
```json
{
  "status": "В обработке"
}
```

**Response (завершено):**
```json
{
  "status": "Завершен",
  "result": {
    "СтрокаJson": "{...}",
    "РаспознанныйТекст": "...",
    "ОписаниеОшибки": ""
  }
}
```

**Response (ошибка):**
```json
{
  "status": "Ошибка",
  "result": {
    "ОписаниеОшибки": "Описание ошибки"
  }
}
```

## Risks / Trade-offs

**Risk 1: API Timeout**
- Mitigation: Установить разумный таймаут (60 сек) и показать понятное сообщение
- Fallback: Предложить пользователю повторить попытку позже

**Risk 2: API Rate Limiting**
- Mitigation: Пока не реализовано, но в будущем добавить кэширование результатов
- Impact: Пользователь может получить ошибку при частых запросах

**Risk 3: Большой размер текста решения**
- Mitigation: API должен обрабатывать большие тексты, но можно добавить ограничение на клиенте
- Impact: Очень длинные решения могут обрабатываться дольше

**Trade-off: Polling vs WebSocket**
- Выбрали polling для простоты реализации
- WebSocket был бы эффективнее, но требует дополнительной инфраструктуры
- Polling достаточно для текущих требований (1 запрос каждые 2-5 секунд)

## Migration Plan

**Phase 1: Backend Setup**
1. Добавить переменные окружения в `.env`
2. Создать proxy endpoints в `server.js`
3. Протестировать с curl/Postman

**Phase 2: API Service**
1. Добавить типы в `types.ts`
2. Реализовать функции в `services/api.ts`
3. Протестировать с реальным API

**Phase 3: UI Implementation**
1. Добавить состояние и обработчики в `Modal.tsx`
2. Создать компоненты для отображения результатов
3. Стилизовать в соответствии с дизайном

**Phase 4: Testing & Refinement**
1. Тестирование с реальными данными
2. Обработка edge cases
3. UX улучшения

**Rollback Plan:**
- Кнопка "AI Анализ" уже существует, но ничего не делает
- Если что-то пойдет не так, можно просто не показывать результаты
- Откат изменений в `server.js` и `services/api.ts` не повлияет на существующий функционал
