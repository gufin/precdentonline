# Design: Исправление семантического поиска

## Context
Проект "Прецедент Онлайн" использует два вида поиска:
1. **Классический поиск** - по ключевым словам через API ГАС Правосудие
2. **Семантический поиск** - по смыслу запроса через внешний сервис semsearch.ru

Семантический поиск был изначально настроен на IP-адрес `http://195.35.56.180`, но должен использовать доменное имя `https://semsearch.ru/` для правильной работы HTTPS и сертификатов.

### Проблемы
1. Используется HTTP вместо HTTPS
2. Используется IP адрес вместо доменного имени
3. Фильтры не полностью соответствуют API спецификации (отсутствует поддержка `categories`)
4. Нет чёткого разделения фильтров для классического и семантического поиска

## Goals / Non-Goals

### Goals
- Исправить URL семантического API на `https://semsearch.ru/`
- Добавить поддержку фильтра `categories` в типах
- Обеспечить стабильное подключение к API
- Документировать различия в фильтрах между двумя типами поиска

### Non-Goals
- Изменение UI для добавления селектора категорий (будет в отдельном change)
- Изменение логики фильтрации в классическом поиске
- Миграция с классического на семантический поиск

## Decisions

### Decision 1: Использование доменного имени вместо IP
**Выбор:** `https://semsearch.ru/` вместо `http://195.35.56.180`

**Обоснование:**
- HTTPS обеспечивает безопасность передачи данных (API ключи)
- Доменное имя более устойчиво к изменениям инфраструктуры
- Упрощает работу с SSL сертификатами
- Соответствует best practices для production окружения

**Альтернативы:**
- Оставить IP адрес - отклонено из-за проблем с HTTPS
- Использовать и IP, и домен с fallback - излишнее усложнение

### Decision 2: Добавление поля categories в типы
**Выбор:** Добавить `categories?: string[] | null` в `SearchFilters` и `SearchResultItem`

**Обоснование:**
- API поддерживает фильтрацию по категориям дел
- Категории важны для юридического поиска (гражданские/административные/уголовные дела)
- Массив позволяет фильтровать по нескольким категориям одновременно
- API возвращает категории в результатах

**Альтернативы:**
- Игнорировать categories - теряем важную функциональность API
- Использовать строку вместо массива - не соответствует API спецификации

### Decision 3: Разделение фильтров
**Выбор:** Документировать, но пока не разделять на уровне типов

**Обоснование:**
- FilterState используется для UI фильтрации после получения результатов
- SearchFilters используется для запросов к семантическому API
- Классический API не использует структурированные фильтры
- Разделение на уровне типов усложнит код без явной выгоды

**Будущее улучшение:**
- Создать отдельные типы `ClassicSearchFilters` и `SemanticSearchFilters`
- Добавить валидацию фильтров перед отправкой запроса

## API Specification Summary

### Semantic Search API (semsearch.ru)

**Base URL:** `https://semsearch.ru/`

**Endpoints:**
- `POST /api/v1/search` - семантический поиск
- `GET /api/v1/cases/{case_number}` - получение полного текста дела
- `GET /health` - проверка здоровья сервиса

**Authentication:** 
- Header: `X-API-Key: <API_KEY>`

**Supported Filters:**
```typescript
interface SearchFilters {
  date_from?: string | null;    // ISO 8601: YYYY-MM-DD
  date_to?: string | null;      // ISO 8601: YYYY-MM-DD
  court?: string | null;        // Точное совпадение названия суда
  doctype?: string | null;      // Тип документа (Решение, Постановление, etc.)
  categories?: string[] | null; // Категории дел (OR условие)
}
```

**Request:**
```typescript
{
  query: string;           // Текст запроса (обязательно)
  limit?: number;          // 1-100, default: 20
  min_score?: number;      // 0.0-1.0, default: 0.5
  filters?: SearchFilters; // Опциональные фильтры
}
```

**Response:**
```typescript
{
  query: string;
  took_ms: number;
  items: SearchResultItem[];
}

interface SearchResultItem {
  document_id: string;        // UUID
  case_number: string;
  court_name: string | null;
  doc_date: string | null;    // ISO 8601
  score: number;              // 0.0-1.0
  snippet: string;
  highlights: string[];
  url: string;
  doctype: string | null;
  categories?: string[] | null; // Категории дела
}
```

### Classic Search API (api.you-right.ru)

**Base URL:** `https://api.you-right.ru/gas`

**Поддерживает:**
- Поиск по ключевым словам
- Поиск по номеру дела
- Фильтрация происходит на клиенте после получения результатов

## Risks / Trade-offs

### Risk 1: Изменение URL может сломать существующие запросы
**Mitigation:**
- Оба адреса (IP и домен) указывают на один сервер
- Проверить подключение в dev/staging до деплоя в prod
- Сохранить старый URL в комментариях для rollback

### Risk 2: SSL сертификат может быть невалидным
**Mitigation:**
- Использовать curl/проверку сертификата перед деплоем
- В случае проблем: использовать только HTTP (не рекомендуется)
- Связаться с владельцами API для исправления сертификата

### Risk 3: API может не поддерживать HTTPS
**Mitigation:**
- Проверено: API работает на HTTPS (тестовый запрос выполнен успешно)
- Fallback не нужен

## Migration Plan

### Phase 1: Code Changes (этот change)
1. Обновить константы URL в коде
2. Добавить поддержку categories в типах
3. Обновить документацию

### Phase 2: Testing
1. Тестирование в dev окружении
2. Проверка всех сценариев поиска
3. Мониторинг ошибок подключения

### Phase 3: Deployment
1. Обновить .env файлы на сервере
2. Перезапустить сервис
3. Мониторинг логов и метрик

### Rollback Plan
Если возникнут проблемы:
1. Изменить `SEMANTIC_API_BASE` обратно на `http://195.35.56.180`
2. Перезапустить сервис
3. Время rollback: < 1 минута
