# Design: GitHub Actions Auto-Deploy

## Context
Проект "Прецедент Онлайн" - это веб-приложение с React frontend и Node.js backend, деплоящееся на FTP хостинг через lftp. В настоящее время деплой выполняется вручную с локальной машины разработчика после каждого изменения.

**Текущий процесс деплоя:**
1. Разработчик вносит изменения
2. Локально запускает `npm run build`
3. Запускает `./deploy.sh` для загрузки на FTP
4. Требуется установленный `lftp` и `.env.deploy` с credentials

**Заинтересованные стороны:**
- Разработчики: упрощение процесса деплоя
- Владелец проекта: быстрая доставка фич в production

**Ограничения:**
- Деплой осуществляется на FTP хостинг (server295.hosting.reg.ru)
- Нет возможности использовать SSH/rsync (только FTP/FTPS)
- Статический хостинг без CI/CD инструментов провайдера
- Необходимо сохранить возможность локального деплоя для экстренных случаев

## Goals / Non-Goals

### Goals
- Автоматический деплой при пуше в ветку `main`
- Безопасное хранение FTP credentials через GitHub Secrets
- Создание резервных копий перед деплоем
- Простая настройка для новых разработчиков
- Прозрачность процесса деплоя (логи, статусы)
- Сохранение возможности локального деплоя

### Non-Goals
- Деплой на других платформах (только текущий FTP хостинг)
- Continuous Integration тесты (нет тестов в проекте пока)
- Деплой preview environments для feature-веток
- Автоматический rollback при ошибках (бэкапы делаются, но откат ручной)
- Миграция на другой хостинг или инфраструктуру

## Decisions

### Decision 1: Использование FTP Deploy Action
**Что:** Использовать готовый GitHub Action `SamKirkland/FTP-Deploy-Action@4.3.3` для загрузки файлов на FTP.

**Почему:**
- Проверенное решение с 2.5k+ stars и активной поддержкой
- Поддерживает FTPS (FTP over TLS/SSL) для безопасной передачи
- Встроенная поддержка incremental deploy (загружает только измененные файлы)
- Автоматическое логирование и обработка ошибок
- Поддержка exclude patterns для исключения ненужных файлов

**Альтернативы рассмотрены:**
- `lftp` в GitHub Actions - требует больше настройки, сложнее дебажить
- Собственный Node.js скрипт с FTP клиентом - избыточная сложность, нужно поддерживать
- WebDAV - не поддерживается текущим хостингом

### Decision 2: Trigger на push в main
**Что:** Workflow запускается автоматически при любом push в ветку `main`.

**Почему:**
- Простая и понятная логика: main = production
- Минимальная задержка между коммитом и деплоем
- Соответствует текущему процессу (деплой после каждого изменения)

**Альтернативы рассмотрены:**
- Manual trigger (workflow_dispatch) - добавит дополнительный шаг, замедлит delivery
- Release tags - избыточно для текущего масштаба проекта
- Pull Request merge - то же самое что push в main, но более сложная настройка

### Decision 3: Деплой только dist/ директории
**Что:** На FTP загружается только содержимое директории `dist/` после сборки.

**Почему:**
- Статический хостинг требует только собранные файлы
- Минимальный размер деплоя (без node_modules, исходников, .git)
- Быстрая загрузка и меньшая нагрузка на FTP сервер
- Безопасность: исходный код и .env не попадают на сервер

### Decision 4: Сохранение локального deploy.sh
**Что:** Скрипт `deploy.sh` остается и продолжает работать для локального деплоя.

**Почему:**
- Fallback на случай проблем с GitHub Actions
- Возможность экстренного hotfix-деплоя
- Не ломает существующий workflow разработчиков
- Может использоваться для деплоя на другие окружения в будущем

### Decision 5: GitHub Secrets для credentials
**Что:** FTP credentials хранятся в GitHub Secrets, а не в коде или .env файлах.

**Почему:**
- Безопасность: credentials не попадают в публичный репозиторий
- GitHub Secrets шифруются и доступны только в workflow
- Стандартная практика для CI/CD
- Легко обновлять без изменения кода

**Secrets которые нужно добавить:**
- `FTP_HOST` - адрес FTP сервера
- `FTP_USERNAME` - имя пользователя FTP
- `FTP_PASSWORD` - пароль FTP
- `FTP_SERVER_DIR` - директория на сервере (например, `/`)

### Decision 6: Использование Node.js 18 в workflow
**Что:** GitHub Actions workflow использует Node.js версии 18.

**Почему:**
- Соответствует версии в `package.json` и `Dockerfile`
- Консистентность между локальной разработкой и CI/CD
- LTS версия с долгосрочной поддержкой

## Risks / Trade-offs

### Risk 1: Credentials в GitHub Secrets
**Риск:** Если репозиторий станет публичным, secrets останутся приватными, но workflow будет виден.

**Митигация:**
- Документировать, что репозиторий должен оставаться приватным
- Использовать Environment secrets вместо repository secrets для дополнительной защиты
- Регулярно ротировать FTP пароль

### Risk 2: Неудачный деплой при ошибке сборки
**Риск:** Если `npm run build` упадет с ошибкой, workflow прервется и деплой не произойдет.

**Митигация:**
- Это желательное поведение! Не деплоить broken build
- GitHub Actions автоматически пометит workflow как failed
- Разработчик получит email уведомление о проблеме
- Добавить линтер и type-check в будущем (не в этом proposal)

### Risk 3: FTP сервер недоступен
**Риск:** Если FTP сервер временно недоступен, деплой упадет.

**Митигация:**
- GitHub Actions имеет встроенные retry mechanisms
- FTP-Deploy-Action имеет таймауты и обработку ошибок
- Можно перезапустить workflow вручную после восстановления сервера
- Локальный `deploy.sh` остается как fallback

### Risk 4: Скорость деплоя
**Риск:** GitHub Actions + FTP может быть медленнее чем локальный деплой.

**Trade-off:** 
- Локальный деплой: быстрее, но требует ручного запуска
- GitHub Actions: немного медленнее (запуск runner ~30 сек), но полностью автоматический
- В итоге: общее время от коммита до деплоя сокращается (не нужно ждать разработчика)

### Risk 5: Отсутствие автоматического бэкапа
**Риск:** `FTP-Deploy-Action` не создает бэкапы на сервере как это делает `deploy.sh`.

**Митигация:**
- FTP-Deploy-Action имеет встроенную опцию `dangerous-clean-slate: false` (по умолчанию), что предотвращает случайное удаление всех файлов
- Файлы перезаписываются инкрементально, старые версии остаются на сервере в папке с тайм-стампом (можно настроить через exclude)
- Можно добавить отдельный step для создания бэкапа через curl/lftp перед деплоем
- Локальный `deploy.sh` продолжает создавать бэкапы при ручном деплое

## Migration Plan

### Phase 1: Настройка GitHub Secrets (ручная работа)
1. Перейти в Settings репозитория → Secrets and variables → Actions
2. Добавить secrets:
   - `FTP_HOST`: `server295.hosting.reg.ru`
   - `FTP_USERNAME`: (из .env.deploy)
   - `FTP_PASSWORD`: (из .env.deploy)
   - `FTP_SERVER_DIR`: `/` (или другая директория)
3. Проверить, что secrets сохранены

### Phase 2: Создание workflow файла
1. Создать `.github/workflows/deploy.yml`
2. Настроить trigger на push в main
3. Добавить steps: checkout, setup node, npm install, npm build, FTP deploy
4. Закоммитить и запушить в feature-ветку (НЕ в main!)

### Phase 3: Тестирование
1. Создать тестовый коммит в main (например, minor change в README)
2. Проверить, что workflow запустился
3. Мониторить логи в GitHub Actions
4. Проверить, что файлы загрузились на FTP
5. Проверить работоспособность сайта https://precedent.online

### Phase 4: Документация
1. Обновить README.md с инструкциями по настройке GitHub Secrets
2. Обновить openspec/project.md секцию Git Workflow
3. Добавить troubleshooting guide для частых проблем

### Rollback Plan
Если GitHub Actions деплой не работает:
1. Использовать локальный `deploy.sh` для срочного деплоя
2. Откатить commit который добавил workflow (если нужно)
3. Восстановить сайт из бэкапа на FTP сервере
4. Отладить проблему в feature-ветке перед повторной попыткой

## Open Questions

### Q1: Нужны ли уведомления в Slack/Telegram о деплое?
**Решение:** Не в этом proposal. Можно добавить позже как отдельную фичу если будет запрос.

### Q2: Деплоить ли при push в feature-ветки?
**Решение:** Нет. Только main = production. Preview environments не входят в scope этого proposal.

### Q3: Добавлять ли автоматические тесты перед деплоем?
**Решение:** Не в этом proposal. Сейчас в проекте нет тестов. Когда появятся - добавим отдельный workflow для CI testing.

### Q4: Нужен ли staging environment?
**Решение:** Не в этом proposal. Текущий хостинг не поддерживает несколько окружений. Можно рассмотреть в будущем при миграции на другой хостинг.
